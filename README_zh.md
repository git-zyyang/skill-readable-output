# skill-readable-output

**把漫长的 AI 对话蒸馏为结构化、可回看的 HTML 页面。**

> 把几小时的讨论变成 5 分钟就能重建思考现场的文档——不是转录，是决策记录。

[English](README.md)

---

## 它做什么

当你和 AI 助手进行了一场长对话——调试系统、设计架构、探索研究方向、做产品决策——洞察会被埋在聊天记录里。三个月后重启项目，你会重新问一遍同样的问题。

这个 Skill 从对话中提取**决策**、**推理过程**和**行动项**，渲染为一个干净的、自包含的 HTML 页面，随时可以回看。

## 快速开始

### Claude Code / Kiro 用户

1. 将 `SKILL-readable-output.md` 复制到项目的 `.claude/skills/` 目录：

```bash
mkdir -p .claude/skills
cp SKILL-readable-output.md .claude/skills/
```

2. 在一次有价值的对话结束后，说：

```
整理一下这次讨论
```

或

```
/readable
```

3. Skill 会问 2-3 个快速问题确认场景，然后生成结构化 HTML 文件并在浏览器中打开。

### 其他 AI 助手

Skill 文件本质是一个结构化 prompt。你可以将其适配到任何支持系统提示词或自定义指令的 AI 助手。核心逻辑（阶段、结构模板、反模式）是可迁移的。

## 功能特性

### 6 种场景模板

| 场景 | 结构 | 适用于 |
|------|------|--------|
| 研究设计推演 | 时间线（问题→方案→否决→选定） | 学术讨论、研究设计 |
| 技术架构决策（ADR） | Context→Decision→Consequences | 系统设计、技术选型 |
| Debug 复盘 | 症状→排查→根因→修复→防护 | 排障过程记录 |
| 产品方向讨论 | 问题→方案→取舍→MVP定义 | 产品/功能讨论 |
| 技术选型对比 | 加权矩阵（维度×候选方案） | 框架/库选择 |
| 行动清单提取 | 优先级排序的 TODO | 快速收尾 |

### 4 种视觉风格

| 风格 | 视觉 | 默认用于 |
|------|------|----------|
| 研究叙事 | 暖白底+时间线+callout | 学术、研究 |
| 技术文档 | GitHub风+代码块+badge | 代码、架构 |
| 产品备忘 | 卡片+标签+进度指示 | 产品、功能 |
| 决策备忘 | 极简黑白+优先级标签 | 快速决策 |

### 智能默认

- 只说"整理一下"不想回答问题 → 自动用「自己回看 + 决策记录 + 标准版」
- 风格自动匹配场景（ADR → 技术文档，等等）
- 体量守卫：内容太薄时会反劝你不要选"完整版"

### 内置质量门控

- **3 点上限**：每份文档不超过 3 个核心洞察（强制优先级排序）
- **砍半测试**："只能保留一半留哪几个？"——剩下的才是真骨干
- **4 问自检**：脱离上下文能看懂？决策+理由都在？不是转录？TODO可执行？
- **反模式清单**：11 种已记录的失败模式

## 输出结构

每个生成的 HTML 包含：

```
┌─────────────────────────────────┐
│ 配置条（场景/深度/风格）          │
├─────────────────────────────────┤
│ TL;DR（2-3句核心结论）           │
├─────────────────────────────────┤
│ 主体内容                         │
│（结构随场景变化）                 │
├─────────────────────────────────┤
│ 行动清单（按优先级排序）          │
├─────────────────────────────────┤
│ 参考资料（链接/文献/文档）        │
├─────────────────────────────────┤
│ 开放问题 + 边界声明              │
└─────────────────────────────────┘
```

## 自定义

### 添加新风格

在 Skill 文件的"CSS 设计 token"部分添加新的 `:root` 变量块。HTML 骨架所有风格共用——只有变量不同。

### 调整文件路径

编辑"文件路径（智能路由）"部分以匹配你的项目结构。默认：

```
学术讨论 → docs/discussions/discussion_{主题}_{日期}.html
技术讨论 → {项目目录}/docs/adr_{主题}_{日期}.html
产品讨论 → {项目目录}/docs/decision_{主题}_{日期}.html
```

### 扩展场景

在"阶段 4 · 选主结构"表格中添加行。每个场景需要：名称、推荐结构模式、可选的默认风格映射。

## 工作原理

```
用户触发 → 阶段1：场景确认（2-3个问题）
         → 阶段1.5：体量自检（防注水守卫）
         → 阶段2：定终点（读者读完得到什么）
         → 阶段3：抓核心点 ≤3个（砍半测试）
         → 阶段4：选结构模板
         → 阶段5：写（TL;DR→正文→TODO→参考→出口）
         → 阶段6：自检4问
         → 输出HTML + 自动在浏览器打开
```

## 什么时候不该用

- 对话不到 5 轮（直接记一句话备忘）
- 纯代码实现，没有决策讨论
- 信息已经在其他系统中结构化了（Jira、Notion、Linear）
- 你想要 markdown 不要 HTML（直接说就行）
- 讨论还在进行中（等结束再触发）
